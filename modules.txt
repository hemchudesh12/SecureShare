4.1 Authentication Module

	The Authentication Module is responsible for securely managing user identity and controlling access to the system. It handles user registration, login, OTP verification, and session management in a structured and security-first manner. During registration, users must provide a valid username, email address, and password. The email address is verified using a time-based One-Time Password (OTP) that is generated using a cryptographically secure random number generator (secrets.randbelow), hashed with SHA-256 before storage, and sent to the user via email. The OTP is valid only for a limited duration and allows a restricted number of verification attempts, ensuring that only legitimate users can complete the registration process. Critically, the user record is not committed to the database until OTP verification succeeds, preventing the creation of unverified accounts. Passwords are never stored in plaintext; instead, they are hashed using bcrypt with a random salt before being written to the database, ensuring that credentials remain protected even in the event of a database breach.

	During login, the system verifies the entered credentials against the stored bcrypt hash and enforces login-type consistency, distinguishing between individual and organization-based logins. Failed login attempts are tracked per user, and accounts are temporarily locked for fifteen minutes after exceeding the configured maximum number of failed attempts, mitigating brute-force attacks. Rate limiting is applied to the login, registration, OTP verification, and key retrieval endpoints to restrict excessive requests from a single source. All login activities, including login type, IP address, timestamp, and success or failure outcome, are recorded in the login log table, enabling ongoing monitoring and detection of suspicious behaviour. Upon successful authentication, user context is stored in an encrypted server-side session, and the user's plaintext password is held temporarily in the session solely to decrypt the RSA private key envelope for file operations, after which it is never persisted elsewhere.


4.2 Organization Management Module

	The Organization Management Module ensures that the system operates within a controlled and isolated structure by grouping users into organizations. Every user in SecureShare is associated with a specific organization, and all file-sharing operations are scoped exclusively within that organization. This boundary prevents unauthorized data access across different groups and guarantees data isolation between tenants. The module allows administrators to create new organizations at registration time or through the admin interface, and automatically assigns admin status and approval to the founding user of each organization.

	When an existing user wishes to join an organization, they submit a join request that is placed in a pending state. An administrator belonging to that same organization must review and approve or reject the request before access is granted. The approval workflow is enforced through the access control layer, which verifies that the approving administrator belongs to the same organization as the request, preventing cross-organization privilege escalation. Administrators can also remove members from the organization, revoking their organizational affiliation and access approval. All administrative actions such as organization creation, join requests, approvals, rejections, and member removals are recorded in the immutable audit log, ensuring full accountability. The module also triggers notification emails to both the administrator and the requesting user at each stage of the workflow, providing real-time communication and transparency throughout the membership management process.


4.3 File Management Module

	The File Management Module is responsible for handling all file-related operations, including upload, encryption, sharing, and download. When a user uploads a file, the system first saves it to a temporary location before processing it through the cryptographic pipeline. The file is encrypted in a chunk-based manner using AES-256-CBC with a freshly generated random key and initialization vector, ensuring that plaintext data is never written to the permanent storage directory. Once encryption is complete, the temporary plaintext file is securely removed from disk using a safe-delete utility. During the upload process, a SHA-256 hash of the file plaintext is computed in a streaming fashion and digitally signed using the sender's RSA private key, producing a cryptographic signature that is stored alongside the file metadata.

	The module allows users to share files with multiple recipients within the same organization. For each recipient, the AES session key is individually encrypted using that recipient's RSA public key via OAEP padding, producing a unique encrypted key entry per user stored in the FileShare table. This ensures that only the intended recipient can decrypt the file using their own private key. During download, the system performs a two-step verification process before any data is transmitted to the client. It first retrieves and decrypts the recipient's AES session key, then recomputes the SHA-256 hash of the file in a streaming, in-memory manner without writing any plaintext to disk, and verifies the stored digital signature against this recomputed hash. Only if the signature is valid does the system proceed to stream the decrypted file directly to the client via a Flask response generator. The module also enforces time-based access expiry and per-user download limits, recording every download event in the access log. This module ensures secure, efficient, and controlled file handling throughout the entire data lifecycle.


4.4 Cryptographic Module

	The Cryptographic Module is the core security component of the system, responsible for implementing all encryption, decryption, key management, and signature operations. It uses a hybrid cryptographic approach that combines the performance of symmetric encryption with the security guarantees of asymmetric encryption. AES-256-CBC is used for encrypting file data because of its high throughput and suitability for large datasets, while RSA-2048 with OAEP padding is used to encrypt the per-file AES session keys, ensuring secure key distribution without requiring a shared secret. All AES keys and initialization vectors are generated using os.urandom, which draws from the operating system's cryptographically secure entropy source.

	The module also implements digital signatures using RSA-PSS with SHA-256. A streaming SHA-256 hash of the file plaintext is computed during upload and signed using the sender's RSA private key. This signature is stored with the file metadata and verified on every download to confirm data integrity and authenticity. If verification fails, the download is aborted immediately. User RSA key pairs are generated at registration time, and private keys are encrypted before storage using a key derived from the user's password via PBKDF2-HMAC-SHA256 with 390,000 iterations, following NIST 2023 recommendations. This prevents unauthorized access to private keys even if the database is fully compromised. The module also provides a streaming decryption generator that yields decrypted plaintext chunks directly to the HTTP response without writing any plaintext to disk at any point, achieving O(1) memory usage regardless of file size. Together, these mechanisms ensure confidentiality, integrity, and authenticity of all data within the system.


4.5 Access Control Module

	The Access Control Module enforces strict authorization policies to ensure that only permitted users can access specific files and perform administrative operations. It acts as a security gatekeeper by evaluating a set of cumulative conditions before granting access to any protected resource. For file access, the module verifies that the requesting user has an active FileShare entry for the requested file, that the share has not been manually revoked by the file owner, that any configured expiry time has not passed, and that the user has not exceeded the permitted download count. If any of these conditions are not satisfied, access is denied immediately and no file data is transmitted.

	For administrative operations, the module verifies that the administrator attempting to approve, reject, or remove a user belongs to the same organization as the request being acted upon. This cross-organization check prevents privilege escalation scenarios where an administrator of one group could interfere with the membership or data of another group. The module is tightly integrated with the file management, routing, and logging systems, ensuring that every access decision is both enforced and recorded. Access denial events are logged with the actor identity and reason, supporting forensic investigation. By implementing these fine-grained, policy-driven access controls, the module prevents unauthorized data exposure and significantly reduces the attack surface of the system.


4.6 Audit and Logging Module

	The Audit and Logging Module is responsible for maintaining a comprehensive and tamper-proof record of all significant system activities. It captures events such as user registrations, logins, login failures, account lockouts, file uploads, file downloads, access denials, key retrievals, key downloads, password changes, organization creation, membership approvals and rejections, and all administrative actions. Each log entry records the actor's user ID, the target resource ID, the organization context, a descriptive action label, additional details, and a precise timestamp, providing a complete and searchable audit trail.

	To ensure the integrity of the log, the module employs a hash-chaining mechanism. Each new log entry is assigned a current_hash value computed as the SHA-256 digest of the concatenation of the previous entry's hash, the current actor ID, action label, details, and timestamp. The previous entry's hash is also stored in the prev_hash field of the new entry, forming a cryptographically linked chain. This makes it computationally infeasible to modify or delete any historical log entry without breaking the chain and making the tampering immediately detectable. The module exposes a verify_log_chain function that replays the entire log in chronological order and recomputes each hash, reporting whether the chain is intact. This verification can be triggered by an administrator through the admin dashboard. The module also supports access transparency by recording the IP address, timestamp, and verification outcome of every file download, allowing file owners and administrators to review who accessed their files and when, enhancing accountability and supporting compliance with security auditing requirements.


4.7 Secure Storage Module

	The Secure Storage Module is responsible for storing encrypted files and all associated metadata in a secure and organized manner. All files are stored exclusively in encrypted form within a designated storage directory on the server. The storage filename assigned to each file is a randomly generated UUID, completely decoupled from the original filename, which prevents enumeration attacks and avoids exposing any information about stored content through file paths. Sensitive metadata such as the original filename, Upload date, file size, AES initialization vector, per-owner encrypted AES key, and RSA-PSS digital signature are stored in the files database alongside the record of each stored file.

	A core design principle of this module is the zero-knowledge architecture, which guarantees that plaintext file data is never written to disk at any point during the upload or download lifecycle. During upload, the file is encrypted in a streaming, chunk-by-chunk fashion before each segment is written to permanent storage, and the temporary unencrypted file is securely deleted immediately after encryption is complete. During download, decryption is performed in a streaming manner using a generator function that produces decrypted chunks on demand and yields them directly to the HTTP response, without materializing the full plaintext in memory or in any temporary file. The chunk size is fixed at 64 kilobytes, ensuring that memory usage remains constant regardless of file size. This approach eliminates the risk of plaintext data residing in temporary files or memory buffers at any point and ensures reliable, scalable, and secure handling of files of arbitrary size.


4.8 Email Notification Module

	The Email Notification Module is responsible for securely delivering transactional emails to users at critical points in the system workflow. It manages two categories of email communication: security-critical OTP delivery, and informational notifications for organization membership events. OTP emails are sent synchronously during the registration and resend-OTP flows, meaning the system waits for delivery confirmation before proceeding with the registration session. This ensures that if email delivery fails, the registration process is halted and the failure is surfaced to the user rather than silently ignored.

	The module enforces a strict set of security guarantees around OTP delivery. The plaintext OTP value is embedded only in the email message body and is never printed to any log file, never returned in any API response, and never passed to any other module after being handed to the email delivery function. The dedicated email logger has log propagation explicitly disabled, preventing any accidental OTP leakage through shared file-based log handlers. All SMTP connections are established using STARTTLS encryption with a verified SSL context, ensuring that email content including the OTP cannot be intercepted in transit. Organization join-request alerts and approval or rejection notifications are delivered asynchronously in background daemon threads using Python's threading module, so these non-critical emails do not block the HTTP request-response cycle. All outbound email credentials are read exclusively from environment variables and are never hard-coded in the source.


4.9 OTP Utility Module

	The OTP Utility Module provides the low-level and session-level primitives used for generating, storing, and verifying One-Time Passwords throughout the authentication and registration workflows. OTPs are generated using secrets.randbelow, the Python standard library's cryptographically secure random number generator, producing a zero-padded six-digit code that avoids predictable sequences. Before any OTP value is stored, whether in the database user record or in the Flask session, it is hashed using SHA-256 so that the plaintext code is never persisted anywhere in the system.

	The module supports two operational modes. In the database-backed mode, OTP fields are written directly to the User ORM object and committed to the database, making the OTP state persistent across requests and server restarts. In the session-backed mode, OTP state is serialized as an ISO-format timestamp dictionary and stored in the Flask session, enabling a stateless pre-registration flow where the user record does not yet exist in the database. Both modes enforce the same security policies: a configurable expiry window, a maximum number of allowed verification attempts with lockout on breach, and a sixty-second resend cooldown period. OTP comparisons are performed using secrets.compare_digest to prevent timing side-channel attacks, ensuring that the comparison takes a constant amount of time regardless of where the mismatch occurs. The module is integrated with the authentication service, authentication routes, and the email notification module to form the complete OTP verification pipeline.


4.10 Configuration and Environment Module

	The Configuration and Environment Module is responsible for centralizing all application settings and loading sensitive credentials securely from the environment. Rather than relying on an external dependency such as python-dotenv, the module implements its own lightweight .env file parser directly within config.py using only Python standard library primitives. At application startup, the parser reads the .env file line by line, strips comments and blank lines, handles optional quote wrapping around values, and injects the parsed key-value pairs into os.environ without overwriting any variables that were already set at the shell level. This design ensures that shell-level environment variables always take precedence over .env file values, which is the expected behaviour in containerized or CI/CD deployment environments.

	All application-wide constants are consolidated in a single Config class, including filesystem paths for the storage directory, the encrypted files directory, and the RSA keys directory. These directories are created automatically at import time using os.makedirs with exist_ok=True, ensuring the storage structure is always in place before any route handler tries to write a file. The Config class also exposes SMTP credentials and server settings for outbound email, OTP policy parameters such as expiry duration and maximum attempt counts, login lockout thresholds, Flask session lifetime and cookie security attributes, and Flask-Limiter rate limit defaults. By importing Config rather than reading os.environ directly, every other module in the project benefits from a single, validated source of configuration truth, making it straightforward to audit or change any setting without searching through multiple files.


4.11 Session and Cookie Security Module

	The Session and Cookie Security Module governs how user session state is created, maintained, and protected across HTTP requests. Flask's built-in server-side session mechanism is used, with sessions signed using the application's SECRET_KEY via HMAC to prevent client-side tampering. Sessions are configured as permanent with a lifetime of thirty minutes of inactivity, after which the session cookie expires and the user must log in again. This limits the window of opportunity for session hijacking in the event that a cookie is stolen.

	Several cookie security attributes are enforced through the Config class. The HttpOnly flag is set to true on all session cookies, preventing client-side JavaScript from reading the cookie value and mitigating cross-site scripting based session theft. The SameSite attribute is set to Lax, which blocks the session cookie from being sent with cross-origin POST requests initiated by third-party sites, providing defence against cross-site request forgery in addition to the explicit CSRF token enforcement provided by Flask-WTF. The Secure flag is configurable and is intended to be set to true in production deployments to ensure cookies are transmitted only over HTTPS connections. The extensions module wires these settings into the Flask application instance at startup through the CSRFProtect and Limiter initializers, ensuring that both CSRF protection and rate limiting are active globally across all registered blueprints from the first request.


4.12 Standalone Signature Verification Module

	The Standalone Signature Verification Module provides users with the ability to explicitly verify the authenticity and integrity of any file they have been granted access to, independently of the download process. This is implemented as a dedicated route that accepts a file identifier and performs the full cryptographic verification pipeline without streaming the file to the client. When invoked, the module first confirms that the requesting user holds an active and valid access grant for the specified file through the access control layer. It then decrypts the user's RSA private key using the session-held password, uses it to unwrap the AES session key from the FileShare record, and performs an in-memory streaming decryption of the stored encrypted file to recompute its SHA-256 hash without writing any plaintext to disk.

	The recomputed hash is then compared against the stored digital signature using the sender's RSA public key via RSA-PSS verification. The result, including the verification outcome, a human-readable status message, the recomputed hash in hexadecimal, the sender's username, the sender's public key, and the upload timestamp, is returned either as a rendered HTML page or as a structured JSON response depending on the client's Accept header. In both cases, a VerificationLog entry is written to the database recording who performed the verification, when it occurred, the outcome, and the IP address of the request. If the signature is found to be invalid, indicating that the file has been modified after upload, this event is additionally written to the audit log as a SIGNATURE_INVALID action, triggering a permanent record for forensic investigation. This module enhances system transparency by giving users direct visibility into the integrity state of the files they receive.
